# Purpose

本文件供 judge 对照使用，不是给考生看的标准答案。判分时不要求逐字匹配，但高分答案应覆盖关键判断点，并体现清晰取舍逻辑。

# Expected Good Answer Shape

一个好答案通常会：
- 明确推荐一个方案，而不是骑墙
- 解释为什么另外两个方案在当前约束下不优
- 体现短期可交付与长期扩展之间的平衡
- 提供分阶段落地路径
- 提前识别权限、工程投入、演示效果等风险

# Key Points

- “2 周内看到成果”是强约束，意味着前期不能选择过慢、过重的路径
- “对内部资料权限敏感”意味着单纯追求快速上线存在风险
- “技术团队只有 2 人”意味着不能选择前期工程投入过高的方案，除非有非常强的回报论证
- “未来 3 个月很可能接入内部知识库”意味着短期方案不能把后续扩展彻底堵死
- 高质量推荐通常会强调“先交付可演示成果，再逐步增强权限与知识接入能力”

# Common Failure Modes

- 只按“速度最快”选 A，却忽略权限敏感和后续内部资料接入
- 只按“长期最好”选 B，却忽略 2 周内需要出成果和团队资源过小
- 说 C 最平衡，但不给具体平衡逻辑
- 不给时间分阶段建议
- 风险和缓解措施写得过于空泛

# Notes

加分项：
- 能提出 8 周的阶段化路线，例如：
  - 第 1–2 周：做可演示最小版本
  - 第 3–5 周：完善安全和权限控制
  - 第 6–8 周：逐步接入内部文档或知识库
- 能明确指出“先证明价值，再逐步增强控制和扩展”的策略

严重问题：
- 完全忽略题目中的关键约束
- 输出格式不合格
- 没有明确结论
